# 강의 계획

## 형식

- 대상: 시니어 개발자
- 권장 기간: 11주
- 회차: 주 1회, 회당 2.5~3시간
- 구성: 개념 강의 40%, 코드 읽기 20%, 실습 30%, 설계 토론 10%
- 기본 언어: Modern JavaScript + TypeScript

## 사전 준비 세션

본 강의에 들어가기 전에 [학습 준비 문서](./00.학습준비.md)를 따라 VS Code, Node.js, pnpm 또는 npm, Git, Vite 기반 React 실습 프로젝트를 준비합니다.

사전 준비 산출물:

- VS Code에서 열린 실습 프로젝트
- `pnpm dev` 또는 `npm run dev`로 실행되는 React + TypeScript 앱
- 브라우저 개발자 도구와 React Developer Tools 확인

## 회차별 구성

| 회차 | 주제 | 핵심 질문 | 실습 |
| --- | --- | --- | --- |
| 1 | Modern JavaScript 1 | ES5 스타일 코드와 현대 JavaScript는 무엇이 다른가? | var/function 기반 코드를 let/const/arrow/destructuring으로 전환 |
| 2 | Modern JavaScript 2 | 배열, 객체, 비동기 코드를 React 친화적으로 어떻게 작성하는가? | map/filter/reduce, immutable update, async/await 실습 |
| 3 | TypeScript Foundation | 타입은 문서인가, 컴파일러가 검사하는 계약인가? | type/interface/union/generic으로 주문 도메인 모델링 |
| 4 | React Mental Model | React는 언제, 왜 다시 렌더링하는가? | 정적 UI를 컴포넌트로 분해 |
| 5 | State Modeling | 어떤 상태를 어디에 둘 것인가? | 폼과 필터 상태 설계 |
| 6 | Effects | Effect는 동기화 도구인가, 이벤트 처리 도구인가? | 불필요한 effect 제거 |
| 7 | Component API | 오래 유지되는 컴포넌트 API는 무엇이 다른가? | 재사용 컴포넌트 설계 |
| 8 | Data Fetching | 서버 상태는 클라이언트 상태와 어떻게 다른가? | CRUD와 캐시 무효화 |
| 9 | Performance | memoization은 최적화인가 부채인가? | Profiler로 병목 찾기 |
| 10 | Testing | 사용자의 신뢰를 테스트로 어떻게 보장할 것인가? | 핵심 플로우 테스트 작성 |
| 11 | Architecture | 팀 규모에서 React 구조는 어떻게 진화해야 하는가? | 아키텍처 리뷰와 개선안 작성 |

## 평가 기준

- 코드가 현대 JavaScript의 스코프, 참조, 비동기 모델을 올바르게 사용하는가
- 타입이 도메인 모델, 함수 계약, 컴포넌트 props를 명확히 설명하는가
- 코드가 React 렌더링 모델과 충돌하지 않는가
- state와 effect의 책임이 명확한가
- 컴포넌트 API가 변경에 견딜 수 있는가
- 성능 개선이 측정에 기반하는가
- 테스트가 구현 세부사항보다 사용자 행동을 검증하는가
- 팀 내 합의 가능한 설계 문서로 설명 가능한가

## 사전 준비

- Node.js LTS
- pnpm 또는 npm
- ES5 JavaScript 경험
- 브라우저 개발자 도구 기본 사용 경험
- TypeScript는 강의 중 필요한 범위에서 함께 도입
